# 日志分析软件 — 需求规格（阶段 1：固定格式日志分析基础版）

## 概述
目的: 为测试人员提供轻量、单机、面向自检与问题定位的日志分析工具。阶段 1 聚焦已经按固定格式生成的日志，不处理日志采集、清洗、标准化和多格式适配。

目标用户: 测试工程师（个位数同时使用），无需登录。

部署环境: 测试人员本机运行的本地 Web 应用，不作为集中式服务器使用；最大内存限制 2GB；单次日志文件最大约 30MB（可以是一个文件，也可以拆分为多个文件）。

关键目标:
- 支持 30MB 级固定格式日志加载、解析、索引和查询。
- 支持关键字、正则、时间范围、等级、应用前缀、tag 等条件查询。
- 支持保存常用查询，并为查询增加描述、分组和标签，方便后续复用。
- 常用操作响应时间 <= 1s；日志较大时允许分批加载，但需要在 5s 内进入可查询或可查看进度的状态。

## 阶段划分
### 阶段 1：固定格式日志分析基础版
包含:
- 本地日志文件加载。
- 按既定日志格式解析字段。
- 基于关键字和正则的日志检索。
- 时间范围、等级、应用前缀、tag 等过滤。
- 查询结果分页、高亮、上下文查看。
- 保存查询、描述、分组和标签。
- 导入必要业务数据，例如保存查询配置、分组/标签配置。
- 查询结果导出 CSV/JSON。

不包含:
- 日志采集、拉取、上传服务。
- 日志清洗、标准化、多格式自动识别。
- 自定义日志格式解析规则。
- 时延分析图表。
- 业务流程节点录入与关联。
- 错误原因解释规则库。
- 多租户认证、权限控制、集中式中间件（如 Elastic 集群）。
- 复杂运维、安全审计、在线多人协作。

### 阶段 2：时延分析
基于阶段 1 的解析结果，按用户配置的阶段边界日志计算时延并绘图。用户可以为某个应用配置阶段链路，例如 `A日志 -> B日志 -> C日志`，其中 `A日志 -> B日志` 的耗时记为 `B阶段`，`B日志 -> C日志` 的耗时记为 `C阶段`。用户选择某次请求后，图中按不同应用分开展示该请求内各阶段耗时；其他同类请求作为统计数据展示，例如请求数、平均耗时、P90、异常请求数、最慢应用、最慢阶段。

阶段 2 的主展示建议:
- 以单次请求为主视图，由用户选择要分析的请求。
- 在选中请求中，按应用前缀分泳道展示各阶段耗时。
- 不同阶段使用不同颜色，例如 `B阶段`、`C阶段`、`D阶段` 分别用不同色块表示。
- 慢阶段或超过阈值的阶段使用强调色展示。
- 右侧或顶部展示其他请求的统计摘要，不把所有请求展开成主图。
- 点击某个阶段后，可以查看该阶段对应的前后边界日志和上下文日志。

阶段 2 的设计需要保持解耦:
- 数据层: 只负责提供阶段边界日志、请求标识、应用前缀、时间戳、原始日志等基础数据。
- 加工层: 负责根据阶段规则把日志加工为请求链路、阶段耗时、应用泳道、统计摘要等展示无关的数据结构。
- 展示层: 只消费加工层输出的数据模型并负责 UI 呈现，例如时间线、泳道、颜色、tooltip、下钻入口。
- UI 调整不应影响日志解析、阶段计算和统计逻辑；后续如果替换图表样式或布局，优先只修改展示层。

### 阶段 3：业务流程与错误解释
支持业务流程/节点录入，节点绑定日志规则；预先维护错误日志描述，命中后直接向用户展示原因解释。

## 日志格式
阶段 1 默认输入日志已经完全符合固定格式，字段都会存在，不做多格式兼容或字段缺失容错。示例:

```log
20675： 06-12 10:39:38.257 32033 32033 W A00010/com.huawei.hmos.aibase/WakeupManager: mainProcess dispatch wakeup request [undefined,]
```

建议解析字段:
- `lineNo`: 日志行号，例如 `20675`
- `timestamp`: 时间戳，例如 `06-12 10:39:38.257`
- `pid`: 进程 ID，例如 `32033`
- `tid`: 线程 ID，例如 `32033`
- `level`: 日志级别，例如 `W`
- `appPrefix`: 应用前缀，例如 `A00010`
- `packageName`: 包名，例如 `com.huawei.hmos.aibase`
- `tag`: tag，例如 `WakeupManager`
- `message`: 日志消息主体
- `raw`: 原始日志行

阶段 1 全部按单行日志处理，不做多行异常堆栈合并。解析失败的行需要保留原始内容，并在加载结果中展示失败数量与失败样例，避免静默丢失日志。

## 功能需求（阶段 1）
### 日志加载
支持用户选择本地单个或多个日志文件，按行读取并解析为结构化日志条目。

验收标准:
- 常规日志加载并完成初次索引 <= 1s。
- 当日志较大或文件较多时，支持分批加载，并在 5s 内进入可查询或可查看加载进度的状态。
- 峰值内存 < 2GB。
- 加载完成后展示总行数、成功解析行数、解析失败行数。

### 固定格式字段解析
按既定日志格式抽取 `lineNo`、`timestamp`、`pid`、`tid`、`level`、`appPrefix`、`packageName`、`tag`、`message`、`raw`。

验收标准:
- 对符合示例格式的日志解析正确率 >= 99%。
- 解析失败行不会中断整体加载。
- 用户可以查看解析失败样例。

时间处理:
- 日志时间戳不包含年份，阶段 1 不补充年份。
- 时间过滤和排序仅基于当前加载日志内的 `MM-DD HH:mm:ss.SSS`。
- 阶段 1 不处理跨年日志场景。

### 关键字与正则搜索
支持关键字、短语和正则表达式搜索。搜索范围默认包含 `raw` 和 `message`，并支持限定到指定字段。

验收标准:
- 普通关键字查询返回首页 <= 1s（90pct）。
- 正则查询返回首页 <= 1s（90pct，基于 30MB 日志和典型正则）。
- 查询结果支持命中内容高亮。
- 查询结果支持分页，避免一次性渲染大量日志。

约束:
- 普通关键字优先使用索引或缓存结果。
- 正则搜索可以在当前过滤条件命中的候选范围内执行，避免对全量日志进行重复低效扫描。
- 对明显高风险或执行过慢的正则，需要有超时或中断机制。

### 条件过滤
支持按时间范围、日志级别、应用前缀、包名、tag 过滤日志。

验收标准:
- 条件可以与关键字/正则组合使用。
- 清空条件后能恢复完整结果集。
- 时间范围过滤基于解析出的 `timestamp` 字段。

### 保存查询与描述
支持保存常用查询，并为查询增加名称、描述、分组和标签，方便后续直接复用。

保存内容:
- `name`: 查询名称
- `description`: 查询说明
- `group`: 查询分组
- `tags`: 查询标签
- `keyword`: 关键字或正则表达式
- `searchMode`: 关键字/短语/正则
- `timeRange`: 时间范围
- `level`: 日志级别过滤
- `appPrefix`: 应用前缀过滤
- `packageName`: 包名过滤
- `tag`: tag 过滤
- `caseSensitive`: 是否大小写敏感
- `createdAt`: 创建时间
- `updatedAt`: 更新时间

验收标准:
- 保存后可以在 UI 中查看、调用、编辑、删除。
- 保存查询支持按分组和标签筛选。
- 调用保存查询后自动恢复对应查询条件。
- 保存查询持久化到本地配置文件，重启后仍可使用。

### 业务数据导入
支持导入阶段 1 使用所需的本地业务数据，主要包括保存查询、查询分组和标签配置。

验收标准:
- 支持从本地 JSON/YAML 文件导入保存查询配置。
- 导入后可以直接在 UI 中使用对应查询、分组和标签。
- 导入失败时给出明确错误提示，不影响已加载日志。

### 查询结果查看
搜索结果以列表形式展示，优先展示用户定位问题最需要的信息。

建议展示字段:
- 时间
- 级别
- 应用前缀
- tag
- 消息主体
- 原始行号

验收标准:
- 支持查看完整原始日志行。
- 支持查看当前命中日志前后若干行上下文。
- 大结果集分页或虚拟滚动展示，不造成界面卡顿。

### 导出
支持导出当前查询结果。

验收标准:
- 查询结果可导出为 CSV/JSON。
- 导出内容包含结构化字段和原始日志行。
- 导出文件可以被常见表格工具或文本工具打开。

## 非功能需求
- 性能: 常规日志加载、查询和保存查询等常用操作 <= 1s；日志较大或文件较多时支持分批加载，并在 5s 内进入可查询或可查看进度的状态。
- 资源: 运行时峰值内存 <= 2GB；避免每次查询重复读取磁盘。
- 可靠性: 解析失败、正则错误、文件读取失败都需要给出明确提示。
- 可维护性: 保存查询和本地配置使用 JSON/YAML 存储。
- 可观测性: 展示加载耗时、索引耗时、查询耗时、解析失败数量等基础指标。

## 实现建议
- 采用本地 Web 应用形态，由测试人员在本机启动并通过浏览器访问，不依赖集中式服务器。
- 加载阶段一次性读取并解析日志，构建内存索引或轻量缓存，减少重复磁盘 IO。
- 日志较大或多个文件加载时，采用分批加载和渐进索引，优先让用户尽快看到进度并开始查询已处理的数据。
- 查询层统一处理关键字、正则和结构化过滤条件。
- 正则搜索需要超时保护，避免复杂表达式影响整体体验。
- UI 首屏优先展示文件加载、查询条件、保存查询入口、查询分组/标签入口和结果列表；非核心信息异步加载。

## 测试与验收计划
- 准备 30MB 固定格式 sample log。
- 单元测试: 固定格式解析器、查询条件组合、保存查询持久化、导出。
- 性能测试: 加载 30MB 日志并执行典型关键字查询、典型正则查询、组合过滤查询，记录 50/90/99pct 延迟与峰值内存。
- 用户验收场景:
  1. 加载日志并通过关键字定位目标日志。
  2. 使用正则搜索一类错误日志。
  3. 保存一个带分组和标签的常用查询，重启后再次调用。
  4. 按时间范围和应用前缀过滤日志并导出结果。

- 示例日志: 30MB 固定格式 sample log 或生成脚本。
- 示例业务数据: 保存查询、查询分组和标签配置。
- 测试脚本: 单元测试与性能测试。

## 已确认决策
1. 阶段 1 日志格式完全固定，按当前示例格式解析。
2. 时间戳不包含年份，阶段 1 不补充年份，只在当前加载日志内按 `MM-DD HH:mm:ss.SSS` 排序和过滤。
3. 阶段 1 全部按单行日志处理，不做多行异常堆栈合并。
4. 保存查询支持简单列表、分组和标签；阶段 1 不做置顶排序。
5. 阶段 1 采用本地 Web 应用形态，测试人员在本机打开使用，不作为服务器部署。
6. 导入必要业务数据后即可使用，阶段 1 的业务数据主要是保存查询、查询分组和标签配置。
7. 常用操作需要在 1s 内响应；日志较大时允许分批加载，但需要在 5s 内进入可查询或可查看进度的状态。
