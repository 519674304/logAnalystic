# 业务请求识别需求

## 需求列表

- REQ-REQUEST-001: 日志加载完成后，系统根据当前规则集自动识别请求。
- REQ-REQUEST-002: 一次请求由全局开始日志标志创建。
- REQ-REQUEST-003: 结束日志可能不存在。
- REQ-REQUEST-004: 如果存在结束日志，请求在第一条匹配的结束日志处结束，并记录结束结果。
- REQ-REQUEST-005: 如果没有结束日志，请求分析到下一次全局开始日志之前。
- REQ-REQUEST-006: 用户可以按业务流程、时间、结束结果、是否异常筛选请求列表。
- REQ-REQUEST-007: 用户可以选择某一次请求进入时延分析视图。
- REQ-REQUEST-008: 请求范围内的成功解析日志都归属于该请求。
- REQ-REQUEST-009: 应用前缀只用于过滤和展示分组，不用于自动推断业务流程或阶段。
- REQ-REQUEST-010: 一次请求可能多次进入同一个应用，系统以规则定义的阶段位置识别，不以应用前缀变化推断。

## 主流程

1. 系统完成日志解析和时间合并。
2. 系统读取当前规则集中的业务流程定义。
3. 系统按全局开始日志标志划分请求。
4. 系统寻找请求结束日志。
5. 结束日志存在时，记录结束时间和结束结果。
6. 结束日志不存在时，请求范围截止到下一次开始日志之前。
7. 系统生成请求列表。
8. 用户筛选请求列表并选择某一次请求。

## 验收标准

- 给定存在开始日志的日志片段，系统可以形成请求。
- 给定存在结束日志的请求，系统可以记录结束结果。
- 给定无结束日志但存在下一次开始日志的请求，系统只分析到下一次开始日志之前。
- 用户可以按业务流程、时间、结束结果、是否异常筛选请求列表。
- 同一个应用在一次请求中出现多次时，可以通过规则阶段位置分别展示。

## 明确排除

- 不通过时间窗口自动切分请求。
- 不通过应用前缀变化自动切分请求。
- 不要求每个应用都有独立开始和结束标志。
