# 时延分析需求

## 需求列表

- REQ-LATENCY-001: 系统根据规则集中明确配置的阶段计算时延，不自行推断阶段。
- REQ-LATENCY-002: 每个时延阶段至少包含阶段名称、阶段类型、开始日志规则、结束日志规则。
- REQ-LATENCY-003: 阶段类型包含应用处理、应用间传递、应用内部。
- REQ-LATENCY-004: 应用处理和应用内部阶段可以指定所属应用。
- REQ-LATENCY-005: 有分支时，阶段可以指定所属分支。
- REQ-LATENCY-006: 请求总耗时按请求范围内的开始和结束边界计算；无结束日志时，请求可分析范围截止到下一次开始日志之前的最后一条请求日志。
- REQ-LATENCY-007: 应用处理阶段用于表达某个应用内从开始到完成的耗时。
- REQ-LATENCY-008: 应用间传递阶段用于表达一个应用完成到另一个应用开始之间的耗时。
- REQ-LATENCY-009: 应用内部阶段用于表达同一应用内部更细粒度的步骤耗时。
- REQ-LATENCY-010: 同一阶段规则在一次请求中重复匹配时，使用第一次匹配。
- REQ-LATENCY-011: 业务流程可以有分支，分支进入后按该分支独立阶段链计算，汇总后进入共同后续阶段。
- REQ-LATENCY-012: 分支入口由明确配置的分支入口日志识别，第一条命中的分支入口决定请求分支。
- REQ-LATENCY-013: 应用内部阶段定义可以按分支不同而不同。
- REQ-LATENCY-014: 系统提供请求总耗时、阶段耗时、同类请求统计、慢阶段和异常请求数量。

## 阶段示例

对于从 A 应用到 B 应用的请求，可以配置为：

```text
请求开始 -> A 应用完成 = A 处理阶段
A 应用完成 -> B 应用开始 = A 到 B 传递阶段
B 应用开始 -> B 应用完成 = B 处理阶段
```

对于 `a日志 -> b日志 -> c日志` 的场景：

```text
a日志 -> b日志 = b阶段
b日志 -> c日志 = c阶段
```

阶段命名由规则集定义。颜色、布局和图表样式属于展示需求，不影响时延计算规则。

## 主流程

1. 用户选择某一次请求。
2. 系统读取该请求所属业务流程和命中分支。
3. 系统按规则集中的阶段顺序匹配阶段边界日志。
4. 系统计算每个阶段的开始时间、结束时间和耗时。
5. 系统汇总请求总耗时和各类阶段耗时。
6. 系统生成与展示无关的时延分析结果。
7. 展示层消费分析结果，绘制阶段视图和统计摘要。

## 验收标准

- 给定明确阶段规则，系统可以计算阶段耗时。
- 阶段结果包含阶段名称、阶段类型、起止日志、耗时、所属应用和所属分支。
- 应用间传递耗时可以在图上独立展示。
- 有分支请求按命中分支计算，不展开未命中分支。
- 同类请求统计不替代单次请求明细，用户选择的请求始终是主视图。
